# 生产安全基线设计

## 目标

在不重复已经存在的业务功能前提下，建立可安全发布的最小生产基线：数据库演进可审计，关键权限与输入边界有自动化证据，采集器不越过网络安全边界，行为事件在基础设施故障时不会丢失并可持续重试，发布流程能复现验证。

## 范围

本阶段只覆盖以下交付物：

- 将当前 Prisma schema 固化为首个受版本控制的 MySQL migration；生产数据库改用 `prisma migrate deploy`，不再以 `prisma db push` 作为发布步骤。
- 增加统一的 Vitest 测试入口，并为注册、资源隔离、后台授权、采集 URL 安全和 outbox 重试建立测试。
- 审计并加固采集 HTTP 请求：仅 HTTPS、拒绝 URL 凭据及本地/私有网络地址、重定向逐跳验证、响应体限制、超时、页面数和任务时长限制。
- 明确 outbox 的单一自动重试运行方式，提供健康检查和可观测的积压指标；Kafka 不可用时保留待投递事件而不影响主请求。
- 固化发布验证：测试、lint、production build、migration 检查与关键受保护路由的 HTTP 验收。

不包含新页面、搜索和后台功能扩展、播放器体验改造、推荐算法修改、Spark 作业开发或视觉重构。这些内容在基线稳定后分阶段进行。

## 方案比较

### 方案 A：仅补文档与人工发布清单

成本最低，但不能阻止未授权访问、SSRF 回归和事件积压；生产质量仍依赖人工操作。不采用。

### 方案 B：以应用内 API 触发维护任务

可复用现有管理 API，但依赖运营人员或外部请求持续触发重试，失败后容易形成不可见积压。仅保留作人工恢复入口。

### 方案 C：版本化迁移 + 测试边界 + 独立维护进程

应用请求只负责写 outbox；独立 worker/cron 定期投递并暴露健康状态，生产发布由 migration 与 CI 门禁保证。它新增少量部署复杂度，但将安全和可靠性责任明确分离。采用此方案。

## 架构与数据流

### 数据库发布

`prisma/schema.prisma` 是模型唯一来源。以当前 schema 生成首个 migration，提交 `prisma/migrations`。本地开发可使用 migration 开发命令；部署环境仅运行 `prisma migrate deploy`。发布流水线在启动应用前执行迁移，迁移失败则阻止新版本启动。

### 授权与输入边界

认证路由负责创建普通用户，后台页面和 API 均在服务端检查 `ADMIN`。用户资源路由以会话用户 ID 为唯一查询范围，忽略客户端提供的其他用户标识。对外 API 将在进入 Prisma、fetch 或 Kafka 前进行结构、长度、范围和枚举校验，并用 400/401/403 返回可操作错误。

### 采集请求

采集源 URL 在写入时和每个请求前验证。请求处理器禁止非 HTTPS、凭据、回环、私网、链路本地、未指定和多播目标；DNS 解析后也必须验证全部结果。请求禁用自动重定向，收到 3xx 时仅在验证下一个 Location 后继续，且受重定向次数限制。流式读取响应，在达到大小上限时中断；每页有超时，任务还受页面数、总时长与连续失败阈值限制。错误日志只保留脱敏、截断的样本。

### 可靠事件投递

业务请求在同一数据库操作中创建 outbox 事件并立即返回。worker 按批领取到期 `PENDING`/失败事件，Kafka 成功后标记已发送；失败时增加尝试次数并计算下一次重试时间。投递载荷带稳定事件 ID，消费者据此幂等。worker 是唯一的持续重试机制；管理 API 可用于手动重试，不承担定期调度责任。健康端点返回积压数量、最早积压时间、近期投递失败和 worker 最近心跳，机密信息不出现在响应中。

## 错误处理和运行约束

- 数据库迁移、认证配置或 worker 运行所需环境变量缺失时，进程快速失败并输出不含密钥的诊断。
- Kafka 不可用不得使用户行为请求失败；事件保留在 outbox，健康检查显示降级状态。
- 安全校验失败不得发起任何网络请求；采集单页失败记录受限错误并依据连续失败策略结束任务。
- 测试使用 mock 隔离数据库、认证、DNS、网络和 Kafka，不依赖真实基础设施。

## 验收标准

- 仓库包含可由空数据库及现有数据库部署的 Prisma migration，发布说明使用 `prisma migrate deploy`。
- `npm test` 运行全部 Vitest；注册角色、普通用户资源隔离、管理员页面/API、非法输入、SSRF 与重定向拒绝、outbox 成功/失败/重试均有自动化覆盖。
- 任意采集跳转不能抵达私网、回环或不允许协议；大响应、超时、超页数和超总时长均安全终止。
- Kafka 中断时主请求仍成功写入 outbox；worker 恢复后可发送积压事件，重复运行不会重复产生业务效果。
- `npm test`、`npm run lint`、`npm run build` 通过；发布前 migration 状态和关键 HTTP 路由检查有可执行命令。

## 部署边界

部署环境需要 MySQL、`DATABASE_URL`、认证密钥和应用运行时配置。若启用分析链路，还需 Kafka 和一个持续运行的 outbox worker（进程管理器、容器副本或平台 cron 任选其一，但每个环境只能配置一种权威调度方式）。Spark 不是本阶段的启动前提；其健康状态可在后续分析阶段接入。
